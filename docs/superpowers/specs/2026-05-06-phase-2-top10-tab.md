# Angular 8 + PrimeNG Rewrite — Phase 2 (Top 10 Tab) Design Spec

**Date:** 2026-05-06
**Branch:** `angular-8-rewrite`
**Builds on:** [Phase 1 spec](./2026-05-05-phase-1-overview-tab.md). Tooling, design system, PrimeNG 8.0.3 conventions, FilterStore, `BaseChartComponent`, `<app-form-field>` are all from Phase 1 and inherited.

---

## 1. Why Phase 2

The Overview tab answers *"how is the operation doing right now?"*. Top 10 answers *"who is driving those numbers?"* — three rankings per filter context: airlines, flights, agents.

The original Phase 1 plan claimed Phase 2 would just reuse `HorizontalBarChartComponent`. That's true for **airlines** (already shipped). For **flights** the DTO carries fulfillment context (serviced + requested), and for **agents** the DTO carries 8 useful fields. Honestly visualizing the data means extending the bar chart for flights and using a PrimeNG table for agents.

---

## 2. Scope

**In:**
- A new `/dashboard/top10` route that mounts `Top10TabComponent` inside the Phase 1 `DashboardComponent` shell.
- Tab navigation between Overview and Top 10 in the dashboard shell.
- Three ranking sections: Top 10 Airlines (reuse), Top 10 Flights (stacked bar), Top 10 Agents (table).
- Drill-down: flight click sets `FilterStore.flight`; agent click sets `FilterStore.agentNo`. Single-value filters (toggleable — click same value to clear).

**Out (later phases):**
- Per-tab saved views (Phase 6).
- Sankey on Service Breakup (Phase 3).
- Heatmap on Fulfillment (Phase 4).
- Agent-service matrix on Insights (Phase 5).

## 3. Resolved OQs (decided 2026-05-06)

| OQ | Decision | Why |
|---|---|---|
| **OQ-P2-1** Top Flights chart type | **B — extend HorizontalBarChartComponent with optional secondary series → stacked bar (serviced primary + requested-gap lighter)** | Fulfillment context per flight is the entire reason ops cares about the flight ranking. Dropping it (option A) would lie about the data; a separate stacked-chart wrapper (option C) would diverge over time. ~30 extra lines on the existing component. |
| **OQ-P2-2** Top Agents display | **B — table with 8 columns** | Bar chart can show 1 metric; the agent DTO has 8 useful ones (rank, agent #, name, PRM count, avg duration, top service, days active, avg/day). Table is the honest shape. PrimeNG `<p-table>` already in the dependency tree. |
| **OQ-P2-3** Drill-down on flights/agents | **Yes for both.** Flight click → `FilterStore.flight`; agent click → `FilterStore.agentNo` | Same UX as airline drill-down established in OQ-P1-2; no new patterns. Both filters are single-value (not arrays) — click the currently-selected value to clear. |
| **OQ-P2-4** Layout | **A — vertical stack** | Three tall sections read top-down. Side-by-side wastes width on agents (8-col table). Sub-tabs (option C) hide content unnecessarily for three sections. Revisit if the page feels long after smoke. |

## 4. Backend reality (read first — don't invent)

Three endpoints, all under `/api/prm/rankings/`, all accept the standard `PrmFilterParams` query shape. Already implemented; verified in [`backend/src/PrmDashboard.PrmService/Controllers/RankingsController.cs`](../../../backend/src/PrmDashboard.PrmService/Controllers/RankingsController.cs).

| Method | Endpoint | DTO |
|---|---|---|
| `topAirlines(limit=10)` | `GET /prm/rankings/airlines?limit=10` | `RankingsResponse { items: RankingItem[] }` (Phase 1) |
| `topFlights(limit=10)` | `GET /prm/rankings/flights?limit=10` | `FlightRankingsResponse { items: FlightRankingItem[] }` |
| `topAgents(limit=10)` | `GET /prm/rankings/agents?limit=10` | `AgentRankingsResponse { items: AgentRankingItem[] }` |

DTO shapes from [`backend/src/PrmDashboard.Shared/DTOs/RankingDtos.cs`](../../../backend/src/PrmDashboard.Shared/DTOs/RankingDtos.cs) — the C# record is authoritative; on the wire each field is camelCase:

```ts
// Source: backend/src/PrmDashboard.Shared/DTOs/RankingDtos.cs

export interface FlightRankingItem {
  label: string;            // flight number e.g. "AI102"
  servicedCount: number;    // PRM services actually performed
  requestedCount: number;   // PRM services booked/expected
  percentage: number;       // serviced as % of total
}
export interface FlightRankingsResponse { items: FlightRankingItem[]; }

export interface AgentRankingItem {
  rank: number;
  agentNo: string;          // e.g. "AGT-007"
  agentName: string;
  prmCount: number;
  avgDurationMinutes: number;
  topService: string;       // dominant SSR code, e.g. "WCHR"
  topServiceCount: number;
  topAirline: string;
  daysActive: number;
  avgPerDay: number;
}
export interface AgentRankingsResponse { items: AgentRankingItem[]; }
```

## 6. `Top10TabComponent` — orchestrator

```ts
@Component({ selector: 'app-top10-tab', templateUrl: './top10-tab.component.html', styleUrls: ['./top10-tab.component.scss'] })
export class Top10TabComponent implements OnInit, OnDestroy {
  private destroy$ = new Subject<void>();

  loading$ = new BehaviorSubject<boolean>(false);

  // Bar-chart inputs (existing HorizontalBarChartComponent shape)
  topAirlines$ = new BehaviorSubject<BarDatum[]>([]);

  // Stacked-bar inputs — primary = serviced, secondary = the gap to requested
  topFlightsServiced$ = new BehaviorSubject<BarDatum[]>([]);
  topFlightsGap$      = new BehaviorSubject<BarDatum[]>([]);

  // Table input
  topAgents$ = new BehaviorSubject<AgentRankingItem[]>([]);

  constructor(public filters: FilterStore, private data: PrmDataService) {}

  ngOnInit(): void {
    this.filters.queryParams$.pipe(
      debounceTime(50),
      switchMap(() => {
        if (this.filters.airportSnapshot.length === 0 || !this.filters.dateFromSnapshot) { return EMPTY; }
        this.loading$.next(true);
        return forkJoin({
          airlines: this.data.topAirlines(10),
          flights:  this.data.topFlights(10),
          agents:   this.data.topAgents(10),
        });
      }),
      takeUntil(this.destroy$),
    ).subscribe(
      r => {
        this.topAirlines$.next((r.airlines.items || []).map(a => ({ label: a.label, value: a.count })));

        // Map FlightRankingItem → two parallel BarDatum series
        const flights = r.flights.items || [];
        this.topFlightsServiced$.next(flights.map(f => ({ label: f.label, value: f.servicedCount })));
        this.topFlightsGap$.next(flights.map(f => ({
          label: f.label,
          value: Math.max(0, f.requestedCount - f.servicedCount),  // never negative
        })));

        this.topAgents$.next(r.agents.items || []);
        this.loading$.next(false);
      },
      err => { console.error('[top10] forkJoin failed', err); this.loading$.next(false); },
    );
  }

  ngOnDestroy(): void { this.destroy$.next(); this.destroy$.complete(); }

  // Drill-down handlers (OQ-P2-3)
  onAirlineBarClick(payload: { category: string; value: number }): void {
    if (payload && payload.category) { this.filters.toggleAirline(payload.category); }
  }
  onFlightBarClick(payload: { category: string; value: number }): void {
    if (payload && payload.category) { this.filters.toggleFlight(payload.category); }
  }
  onAgentRowClick(agent: AgentRankingItem): void {
    if (agent && agent.agentNo) { this.filters.toggleAgentNo(agent.agentNo); }
  }
}
```

Three independent BehaviorSubjects per chart instead of one combined view-model — same pattern as `OverviewTabComponent`. The `Math.max(0, requested - serviced)` guard exists because some CSV rows can theoretically have `serviced > requested` after manual edits; we'd rather show "fully serviced" than a negative bar.

## 7. `HorizontalBarChartComponent` extension

New optional `@Input() secondaryData?: BarDatum[]`. When present and non-empty, the chart switches to **stacked** mode with two echarts series sharing the same y-axis categories.

```ts
@Input() data: BarDatum[] = [];                   // existing — primary (serviced)
@Input() secondaryData?: BarDatum[];              // NEW — secondary (gap to requested)
@Input() primaryLabel = 'Serviced';               // NEW — legend label
@Input() secondaryLabel = 'Requested (gap)';      // NEW — legend label
```

Stacked-mode options diff:

```ts
series: [
  {
    name: this.primaryLabel,
    type: 'bar',
    stack: 'rank',
    data: this.topRows.map(d => d.value),
    barMaxWidth: 24,
    itemStyle: { color: resolvePrimary() },
    emphasis: { focus: 'series' },
  },
  ...(this.secondaryData ? [{
    name: this.secondaryLabel,
    type: 'bar',
    stack: 'rank',
    data: this.secondaryRows.map(d => d.value),
    barMaxWidth: 24,
    itemStyle: { color: resolvePrimary(), opacity: 0.30 },   // same hue, faded
    emphasis: { focus: 'series' },
  }] : []),
],
legend: this.secondaryData ? { data: [this.primaryLabel, this.secondaryLabel], right: 0, top: 0 } : undefined,
```

Existing single-series usage (Overview's Top Airlines) is unchanged: when `secondaryData` is undefined, the component renders exactly as it did in Phase 1. The drill-down click handler (`onChartClick` → `barClick`) also unchanged — clicking either segment of a stacked bar emits the same `{category, value}` payload (echarts surfaces the y-axis category on click regardless of which series is hit).

## 8. Agent table

PrimeNG `<p-table>` (TableModule). Eight visible columns:

| Col | Header | Source field | Format |
|---|---|---|---|
| 1 | # | `rank` | mono int |
| 2 | Agent | `agentNo` | mono |
| 3 | Name | `agentName` | sans |
| 4 | PRM | `prmCount` | mono right-aligned |
| 5 | Avg Duration | `avgDurationMinutes` | mono `XX min` |
| 6 | Top Service | `topService` (`topServiceCount`) | sans + mono count |
| 7 | Top Airline | `topAirline` | sans |
| 8 | Avg / Day | `avgPerDay` | mono `XX.X` |

Each row is clickable (drill-down). Active row (when `agentNo === filters.agentNoSnapshot`) gets `is-selected` styling — soft primary background, primary left-edge accent (mirrors the KPI-card stripe).

```html
<p-table [value]="(topAgents$ | async) || []" [rows]="10" responsiveLayout="scroll" styleClass="agents-table">
  <ng-template pTemplate="header">
    <tr>
      <th class="num">#</th>
      <th class="mono">Agent</th>
      <th>Name</th>
      <th class="num">PRM</th>
      <th class="num">Avg Duration</th>
      <th>Top Service</th>
      <th>Top Airline</th>
      <th class="num">Avg / Day</th>
    </tr>
  </ng-template>
  <ng-template pTemplate="body" let-agent>
    <tr (click)="onAgentRowClick(agent)" [class.is-selected]="agent.agentNo === (filters.agentNo$ | async)">
      <td class="mono num">{{ agent.rank }}</td>
      <td class="mono">{{ agent.agentNo }}</td>
      <td>{{ agent.agentName }}</td>
      <td class="mono num">{{ agent.prmCount | number }}</td>
      <td class="mono num">{{ agent.avgDurationMinutes | number:'1.0-0' }} min</td>
      <td>{{ agent.topService }} <span class="mono muted">({{ agent.topServiceCount }})</span></td>
      <td>{{ agent.topAirline }}</td>
      <td class="mono num">{{ agent.avgPerDay | number:'1.1-1' }}</td>
    </tr>
  </ng-template>
  <ng-template pTemplate="emptymessage">
    <tr><td colspan="8" class="empty">No agents match current filters</td></tr>
  </ng-template>
</p-table>
```

Sorting / pagination are deferred — 10 rows max per call so neither is needed yet.

## 9. `DashboardComponent` tab nav

Currently the dashboard shell renders only the filter bar + `<router-outlet>`. Phase 2 adds inline tab navigation between filter bar and outlet:

```html
<div class="dashboard-shell">
  <app-filter-bar></app-filter-bar>
  <nav class="dashboard-tabs" aria-label="Dashboard sections">
    <a routerLink="overview" routerLinkActive="is-active" queryParamsHandling="preserve">Overview</a>
    <a routerLink="top10"    routerLinkActive="is-active" queryParamsHandling="preserve">Top 10</a>
  </nav>
  <main class="dashboard-content">
    <router-outlet></router-outlet>
  </main>
```

`queryParamsHandling="preserve"` is critical: clicking a tab must NOT clear the active filter set. The router will keep `?airport=...&date_from=...` intact across the navigation.

## 10. `FilterStore` additions

Two new toggle mutators for the drill-down. `flight` and `agentNo` are single-value strings (per the existing `FilterState` shape — see Phase 1 spec §5).

```ts
/** Drill-down: flight bar click. Single-value — clicking same value clears. */
toggleFlight(value: string): void {
  this._flight$.next(this._flight$.value === value ? '' : value);
}

/** Drill-down: agent row click. Single-value. */
toggleAgentNo(value: string): void {
  this._agentNo$.next(this._agentNo$.value === value ? '' : value);
}
```

The single-value semantics differ from `toggleAirline` / `toggleService` (which work on arrays). The shape matches `FilterState.flight: string` and `FilterState.agentNo: string` already defined in Phase 1.

Spec tests cover: setting → toggleFlight(same) clears; toggleFlight(different) replaces; same for agentNo.

## 13. Risks / latent concerns

- **R-P2-1 — Stacked legend rendering.** echarts 4 stacked-bar legends sometimes overlap with the chart on narrow widths. If the smoke test shows clipping, move legend to `bottom: 0` instead of `top: 0` and adjust grid bottom margin.
- **R-P2-2 — Empty `requested - serviced` gap.** When all flights are 100% serviced, every secondary value is 0 and the secondary series renders as zero-height. echarts handles this without errors but the legend still shows. Acceptable; the legend signals the *intent* even when there's no gap to display.
- **R-P2-3 — Long agent names.** `agentName` is a free-text string; >40 chars will wrap or truncate the table. Use `text-overflow: ellipsis` with `max-width: 200px` on the Name column, with full name in a `pTooltip`.
- **R-P2-4 — Tab nav under narrow widths.** Two tabs are fine on mobile; later phases add 5 tabs total which will overflow the viewport. Defer horizontal scroll handling to Phase 5.
